Why write another beginner tutorial?

Because I like teaching and I need a creative outlet for products that are not feasible in a professional environment, but I think would be fun to create or use.

Because most tutorials cover a well-structured learning path, that I think should not exist "in professional setting".

Because I believe programming is about discovery, not about recreating once-learned patterns.

Last but not least, because I want to become a better writer, so I need a topic to write on and people who may give me feedback.

Fellow beginner tutorial reviews

I felt it would only be fair if I left my article-review here, for any reader that may have prior experience with tutorials, to show you what I care about when writing my own. If you just want to learn programming, you may skip this paragraph.

  1. blog.scottlogic.com - Creating Simple Web Services in Go
    • The application cannot be run anywhere and do anything really, it's just an excercise.
    • func Remove(id int) bool is buggy, because it makes assumptions about system state that were not specified anywhere.
    • The implementation of HandleRequest is not even lazy programming, because it's too creative.
    • Blank lines between all significant lines of code is in my opinion not a good habit to get as a beginner.
    • Errors are not handled at all or are handled incorrectly - sometimes return error values get shadowed and/or ignored.
    • No followup after implementing the in-memory app.
  2. smartbear.com - How to build a web service in 5 minutes with go
    • Writing function stubs first and then trying to fill them in is a mistake in my book, because it forces your focus away from what you are doing and into what artificial mold you are trying to push yourself into. And how are you supposed to know what artificial mold you should even try to fit yourself into when writing your own project?
    • The article is very opinionated, but there is no backing for author's opinions.
    • Leaving a newbie three frameworks to choose from without explanation how they should chose is in my opinion very confusing.
    • Here is what the API looks like: is a huge red flag in my opinion, because usually you need to figure out what the API looks like and it's the most difficult part of programming.
    • The type WebService interface is a really weird place to showcase interfaces, because this interface makes no sense - it adds a lot of boilerplate HTTP request handling, that is very framework-specific in chosen library and would require full refactor when changing said library anyways.
    • // Make sure Body is closed when we are done. - sure, but why?
    • Again, no followup.
  3. go.dev - Tutorial: Developing a RESTful API with Go and Gin
    • Very clean tutorial with each step described in much more details than I plan to.
    • Explains how to achieve an objective, but not how to create such objective for yourself or break big objective into smaller tasks.
    • I would say it's a very good tutorial for people who already know programming and want to start using Go now.
  4. honeybadger.io - A Gentle Introduction to Web Services With Go
    • Good, thorough introduction to standard library HTTP handling.
    • Seems to be aimed towards seasoned programmers.
    • No followup, no actual project built.
  5. go.dev - Writing Web Applications
    • Very good tutorial, the only thing I'm missing is a follow-up walthrough showcasing how to start building an actual product.
  6. medium.com/@anilvedala - Setting up a web service in Go
    • Code formatting is off.
    • Again, third-party packages with no choice justification.
    • Very short, shows only the basics

My vision, mission and values?

There is a thing that's "hidden" from most casual users - namely the trifecta of mission, vision and core values. While this is usually just a bunch of buzzwords for big organizations, it's important to recognize the idea behind this mental excercise. Imagine, that you are bringing a new person to your team and you don't want to babysit them 100% of their working time - how can you quickly teach them what principles should guide their decisions at work? That's right, those three statements.

Because I am a single-person organization right now, my goal in defining those statements is to stay on track of my initial goals when new ideas ineavitable come to my mind and my motivation gets redirected into another venture.

Vision Statement

Make the internet faster, safer and easier to understand for not-yet-engineers.

Mission Statement

Produce software that I would like to use, with codebase that can be read and understood by programming newcommers.

Values

  • The Golden Rule - create articles that I would find interesting, create products that I would find useful
  • Transparency - make my failures visible for others, to reassure everyone that failing is an expected and unavoidable part of doing anything.
  • Understandability - try my best to communicate in an understandable way, accepting that understandability is more important than performance
  • Working on sources - whenever possible, use third-party sources and examples. Avoid making claims without a confirmation that they appear in the real world.
  • Quality over quantity - writing a lot of content is quick and easy. Writing good content is hard and slow. Accept the pace. There is enough content existing already.
  • Money will follow - this is a hobby project. If money ever happens to come my way from it, it should be a bonus, not a core focus. Focusing on monetization should be considered a waste of time as often as possible.

If at any point you notice my content derives from those, please let me know so I can decide if I should change my words or my actions.

Inspiration and battle plan

When I was starting to get interested in programming, the internet was very young. Compatibility issues with Internet Explorer 6 were the focus of most HTML / CSS tutorials. Go didn't exist and most webpages were created in PHP, with JavaScript used exclusively to add interactivity to websites.

My "aha" moment happened, when I registered on an online forum created by my online friend. After registering, said friend has sent me my password in clear text and asked me if it looks familiar. I remember the sheer panic that overloaded my body then and there. But it also made me realize, that he has to have access to whatever is holding the password information and he must be able to read it.

The websites people use now are usually giant social media platforms or centralized entertainment platforms. Dead are the small forums, the fan-made websites in plain HTML. There are wiki scripts, WordPress plugins and Facebook groups. As a side effect, this has left the new generation of software engineers without a starting point. Because once you finish the beginner tutorial and create a TODO list, where to go next? How to broaden your horizons? How to jump from "I understand things written in given programming language" to "I have created this product on my own and I can create one for you (possibly only if you pay me)"?

This tutorial aims to close this knowledge gap.

What is the expected end result?

I would like to recreate services that I have used in my youth, that didn't need an army of engineers back then, and I don't expect them to need an army now. I would like to showcase how they used to work and how they can be recreated today, using the simplest tech stack possible.

The beginning is this blog. If this is the first post you see, I am most likely still working on it. Come on, have a look!
